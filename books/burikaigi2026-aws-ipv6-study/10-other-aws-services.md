そして、AWS 利用者の関心事の 1 つ、「実際にパブリック IP(v4) アドレスは節約できるのか？」について触れます。

外部（インターネット）からアクセスするサービスの場合、とりあえず「Dual stack support」が「Yes」になっていれば、IPv4・IPv6 の両方でアクセスが可能です。

ここ数年で IPv6 をサポートするサービスが増えましたね。

Amazon Bedrock や Aurora DSQL など、「クライアント（ブラウザ）から直接アクセスさせる必要がないサービス」については、「Public endpoints support IPv6」「Private endpoints support IPv6」側のみサポートしているケースも増えている感があります。

## 実際にパブリック IP(v4) アドレスは節約できるのか？

例えば、2025 年に外側トンネルで IPv6 がサポートされた Site-to-Site VPN のように、「接続先が固定的」なサービスについては、IPv6 化によるパブリック IPv4 アドレスの節約は可能でしょう。

また、CloudFront（外側についてはパブリック IPv4 アドレスの課金なし／オリジンがパブリック IPv4 アドレスを持つ場合オリジンに課金される）についてはオリジンを IPv6 化する方法もありますし、IPv6 化とは関係なく VPC オリジンに変えてパブリック IPv4 アドレスを使わない方法もあります。

一方、何らかの事情で CloudFront + IPv6 or VPC オリジンではなく ALB や NLB を外側に置いて使い続ける場合、アクセスしてくるユーザーがすべて IPv6 が利用できる環境であれば良いのですが、そうでなければ ALB や NLB にパブリック IPv4 アドレスは必要です。

サーバー環境ではなくクライアント環境として使う場合も、2025 年 12 月現在の GitHub のようにアクセス先が IPv6 に対応していない場合は NAT ゲートウェイ（または NAT／プロキシ用のインスタンス）が必要なので、結局パブリック IPv4 アドレスは必要です。

したがって、「ユースケースによるが、多少の節約はできても大幅な節約をしてゼロに近付けるのは難しい」のが現状、といえるでしょう。

## このチャプターのまとめ

- ここ数年で IPv6 をサポートするサービスが増えた
- ただしパブリック IPv4 アドレスを節約できるケースは意外と限られる
  - IPv4・IPv6 を併用する必要があるケースがまだ多く残る
